# 보안 아키텍처 설계

## 1. 인증 (Authentication)

### 1.1 JWT 토큰 전략
```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   클라이언트  │────▶│   API 서버   │────▶│    Redis     │
│  (Web/Mobile)│◀────│  (Express)   │◀────│ (토큰 저장)   │
└─────────────┘     └─────────────┘     └─────────────┘
```

| 토큰 | 유효기간 | 저장 위치 | 용도 |
|------|---------|----------|------|
| Access Token | 15분 | Web: 메모리 / Mobile: SecureStore | API 인증 |
| Refresh Token | 7일 | Web: httpOnly Cookie / Mobile: SecureStore | 토큰 갱신 |
| Device Token | 영구 | DB (user_devices) | 디바이스 식별 |

### 1.2 토큰 구조
```json
{
  "sub": "user_id (UUID)",
  "role": "admin | dept_admin | user | guest",
  "dept_id": "department_id",
  "device_id": "device_uuid",
  "permissions": ["approval:read", "messenger:write"],
  "iat": 1700000000,
  "exp": 1700000900
}
```

### 1.3 토큰 갱신 플로우
```
1. Access Token 만료 → 401 Unauthorized
2. 클라이언트가 /auth/refresh 호출 (Refresh Token 포함)
3. 서버: Refresh Token 검증 → Redis에서 유효성 확인
4. 새 Access Token + Refresh Token 발급 (Rotation)
5. 이전 Refresh Token 무효화 (Redis에서 삭제)
```

### 1.4 Refresh Token Rotation
- 매 갱신 시 새 Refresh Token 발급 + 이전 토큰 무효화
- 탈취된 토큰 재사용 감지 시 해당 사용자의 모든 세션 강제 로그아웃
- Token Family 추적으로 replay attack 방어

## 2. 인가 (Authorization)

### 2.1 역할 기반 접근 제어 (RBAC)
```
super_admin ─── 전체 시스템 관리, 모듈 ON/OFF
    │
  admin ─────── 회사 설정, 사용자 관리, 전체 데이터 조회
    │
 dept_admin ─── 부서 관리, 부서원 데이터 조회
    │
   user ──────── 일반 기능 사용
    │
  guest ──────── 제한된 읽기 권한
```

### 2.2 권한 매트릭스
| 리소스 | super_admin | admin | dept_admin | user | guest |
|--------|:-----------:|:-----:|:----------:|:----:|:-----:|
| 사용자 관리 | CRUD | CRUD | R(부서) | R(본인) | - |
| 전자결재 | CRUD+설정 | CRUD | CRUD | CRU | R |
| 메신저 | CRUD+관리 | CRUD | CRUD | CRUD | R |
| CCTV | CRUD+설정 | CRUD | R(할당) | R(할당) | - |
| 근태관리 | CRUD+설정 | CRUD | RU(부서) | RU(본인) | - |
| 작업지시서 | CRUD | CRUD | CRUD(부서) | RU(담당) | - |
| 재고관리 | CRUD | CRUD | CRUD | RU | R |
| 문서관리 | CRUD+설정 | CRUD | CRUD | CRUD | R(공유) |
| 관리자콘솔 | Full | 제한적 | - | - | - |

### 2.3 미들웨어 체인
```typescript
// 요청 처리 순서
app.use(rateLimiter)           // 1. Rate Limiting
app.use(helmet)                // 2. HTTP 보안 헤더
app.use(cors)                  // 3. CORS 정책
app.use(authenticateToken)     // 4. JWT 검증
app.use(checkModuleEnabled)    // 5. 모듈 활성화 확인
app.use(authorizeRole)         // 6. 역할 기반 인가

## 3. 비밀번호 보안

### 3.1 해시 정책
| 항목 | 설정 |
|------|------|
| 알고리즘 | bcrypt |
| Salt Rounds | 12 |
| 최소 길이 | 8자 |
| 복잡도 | 영문 + 숫자 + 특수문자 각 1개 이상 |
| 변경 주기 | 90일 (설정 가능) |
| 이전 비밀번호 재사용 | 최근 5개 불가 |

### 3.2 계정 잠금 정책
```
로그인 실패 5회 → 계정 잠금 15분
로그인 실패 10회 → 계정 잠금 1시간
로그인 실패 20회 → 관리자 해제 필요
```

### 3.3 비밀번호 변경 흐름
```
1. 현재 비밀번호 확인
2. 새 비밀번호 복잡도 검증
3. 최근 5개 비밀번호와 비교 (중복 방지)
4. bcrypt 해시 후 저장
5. 모든 기존 세션 무효화 (선택)
6. 변경 알림 이메일/푸시
```

// API 통신 보안 설정
const apiClient = axios.create({
  baseURL: API_BASE_URL,
  timeout: 30000,
  headers: {
    'Content-Type': 'application/json',
    'X-App-Version': APP_VERSION,
    'X-Device-ID': deviceId,
  },
});

// 인터셉터: 토큰 자동 갱신
apiClient.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      const newToken = await refreshToken();
      error.config.headers.Authorization = `Bearer ${newToken}`;
      return apiClient.request(error.config);
    }
    return Promise.reject(error);
  }
);
```

## 7. WebSocket 보안

### 7.1 인증
```typescript
// Socket.IO 연결 시 JWT 검증
io.use(async (socket, next) => {
  const token = socket.handshake.auth.token;
  try {
    const decoded = jwt.verify(token, JWT_SECRET);
    socket.data.user = decoded;
    next();
  } catch (err) {
    next(new Error('UNAUTHORIZED'));
  }
});
```

## 8. 파일 업로드 보안

### 8.1 업로드 제한
| 항목 | 제한 |
|------|------|
| 최대 파일 크기 | 50MB (일반), 500MB (영상) |
| 허용 확장자 | jpg, png, gif, pdf, docx, xlsx, pptx, mp4, mov |
| MIME 타입 검증 | Magic Bytes 기반 이중 검증 |
| 파일명 | UUID 기반 재명명 (원본명 별도 저장) |
| 저장 경로 | 웹 루트 외부, 직접 접근 불가 |
| 바이러스 스캔 | ClamAV 연동 (선택) |

### 8.2 파일 접근 제어
```typescript
// 파일 다운로드 시 서명 URL 방식
// 1. 클라이언트가 파일 ID로 다운로드 요청
// 2. 서버: 권한 확인 → 서명된 임시 URL 생성 (5분 유효)
// 3. 클라이언트: 서명 URL로 직접 다운로드

app.get('/api/files/:id/download', authenticate, async (req, res) => {
  const file = await prisma.file.findUnique({ where: { id: req.params.id } });
  // 권한 확인
  if (!canAccessFile(req.user, file)) {
    return res.status(403).json({ error: 'FORBIDDEN' });
  }
  // 서명 URL 생성
  const signedUrl = generateSignedUrl(file.path, 300); // 5분
  res.json({ url: signedUrl });
});
```

| 항목 | 구현 |

| 스트림 접근 | JWT + CCTV 권한 검증 |

| 녹화 접근 | 관리자 + 감사 로그 기록 |

### 9.2 PTZ 제어 보안
```
PTZ 잠금 메커니즘:
1. 사용자가 PTZ 제어 요청
2. 해당 카메라 잠금 획득 (Redis Lock, TTL 60s)
3. 이미 잠겨있으면 → "다른 사용자가 제어 중" 알림
4. 제어 완료 또는 타임아웃 → 잠금 해제
5. 모든 PTZ 명령 감사 로그 기록
```
