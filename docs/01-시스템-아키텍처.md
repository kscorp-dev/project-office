# 시스템 아키텍처 설계서

## 1. 시스템 개요

**Project Office**는 사내 업무를 통합 관리하는 그룹웨어 플랫폼이다.
PC(웹)과 모바일(iOS/Android 네이티브 앱)에서 동일한 기능을 제공한다.

### 1.1 핵심 목표
- 전자결재, 메신저, CCTV를 하나의 플랫폼에서 통합 제공
- PC와 모바일에서 동일한 사용자 경험 보장
- 실시간 알림 및 데이터 동기화
- 확장 가능한 모듈형 아키텍처

### 1.2 사용자 유형
| 역할 | 설명 | 권한 범위 |
|------|------|----------|
| 시스템 관리자 | 전체 시스템 관리, 사용자/조직 관리 | 모든 기능 접근 |
| 부서 관리자 | 소속 부서 관리, 결재 승인 | 부서 내 관리 기능 |
| 일반 사용자 | 기본 업무 기능 사용 | 할당된 기능 사용 |
| 게스트 | 제한된 기능 접근 (외부 협력업체 등) | 초대된 기능만 |

## 3. 기술 스택 상세

### 3.1 Backend

| 구분 | 기술 | 선정 이유 |
|------|------|----------|
| Runtime | Node.js 20+ | 비동기 I/O, WebSocket 네이티브 지원 |
| Framework | Express.js | 가볍고 유연한 미들웨어 구조 |
| Language | TypeScript | 타입 안정성, 대규모 프로젝트 유지보수 |
| ORM | Prisma | 타입 안전한 DB 접근, 마이그레이션 관리 |
| 실시간 통신 | Socket.IO | WebSocket + Fallback, Room 관리 |
| 인증 | JWT + Refresh Token | Stateless 인증, 모바일 친화적 |
| 파일 저장 | AWS S3 / 로컬 | 확장 가능한 파일 스토리지 |
| 캐시 | Redis | 세션, 캐시, 실시간 상태 관리 |
| 미디어 서버 | FFmpeg + HLS | CCTV RTSP 스트림 → HLS 변환 |

### 3.2 Database

| 구분 | 기술 | 용도 |
|------|------|------|
| RDBMS | PostgreSQL 15+ | 주 데이터 저장소 (사용자, 결재, 조직, 게시판 등) |
| In-Memory | Redis 7+ | 세션 관리, 캐시, 실시간 메시지 큐, 온라인 상태 |

### 3.3 Web Frontend

| 구분 | 기술 | 선정 이유 |

| Framework | React 18 | 컴포넌트 기반, 생태계 |

## 4. 인증 아키텍처

### 4.1 JWT 기반 인증 플로우

```
┌────────┐                      ┌─────────┐                ┌───────┐
│ Client │                      │  API    │                │  DB   │
└───┬────┘                      └────┬────┘                └───┬───┘
    │                                │                         │
    │  1. POST /auth/login           │                         │
    │  {email, password}             │                         │
    │───────────────────────────────>│                         │
    │                                │  2. 사용자 조회           │
    │                                │────────────────────────>│
    │                                │  3. 사용자 정보 반환      │
    │                                │<────────────────────────│
    │                                │                         │
    │                                │  4. 비밀번호 검증         │
    │                                │  5. JWT 토큰 생성        │
    │                                │     - Access Token (15m) │
    │                                │     - Refresh Token (7d) │
    │                                │                         │
    │  6. 토큰 반환                   │                         │
    │  {accessToken, refreshToken}   │                         │
    │<───────────────────────────────│                         │
    │                                │                         │
    │  7. API 요청                    │                         │
    │  Authorization: Bearer {token} │                         │
    │───────────────────────────────>│                         │
    │                                │  8. 토큰 검증            │
    │                                │  9. 요청 처리            │
    │  10. 응답                      │                         │
    │<───────────────────────────────│                         │
```

### 4.2 토큰 갱신

```
Access Token 만료 시:
1. Client → POST /auth/refresh {refreshToken}
2. Server → Refresh Token 유효성 검증
3. Server → 새 Access Token 발급
4. Server → (선택) Refresh Token Rotation
```

### 4.3 보안 정책
- 비밀번호: bcrypt 해싱 (salt rounds: 12)
- Access Token: 15분 유효
- Refresh Token: 7일 유효, httpOnly 쿠키 또는 Secure Storage
- Rate Limiting: 로그인 시도 5회/분 제한
- 비밀번호 정책: 최소 8자, 영문+숫자+특수문자 조합

## 5. 실시간 통신 아키텍처

### 5.1 WebSocket 네임스페이스 설계

```
Socket.IO Server
├── /messenger          # 메신저 실시간 통신
│   ├── join_room       # 채팅방 입장
│   ├── leave_room      # 채팅방 퇴장
│   ├── send_message    # 메시지 전송
│   ├── typing          # 타이핑 표시
│   ├── read_message    # 읽음 확인
│   └── message_received # 메시지 수신
│
├── /notification       # 알림
│   ├── approval_request  # 결재 요청 알림
│   ├── approval_result   # 결재 결과 알림
│   ├── mention           # 멘션 알림
│   └── system            # 시스템 알림
│
├── /presence           # 온라인 상태
│   ├── online          # 접속
│   ├── offline         # 접속 해제
│   ├── away            # 자리비움
│   └── status_change   # 상태 변경
│
└── /cctv               # CCTV 시그널링
    ├── subscribe       # 카메라 구독
    ├── unsubscribe     # 구독 해제
    └── ptz_control     # PTZ 제어 명령
```

### 5.2 메시지 전달 보장

```
1. Client → 메시지 전송 (with temp_id)
2. Server → DB 저장
3. Server → 수신자에게 전달
4. Server → 발신자에게 ACK (temp_id → real_id)
5. 수신자 → 읽음 확인 이벤트
6. Server → 발신자에게 읽음 상태 업데이트
```

## 6. CCTV 스트리밍 아키텍처

```
┌──────────┐     RTSP      ┌────────────────┐     HLS      ┌──────────┐
│  CCTV    │──────────────>│  Media Server  │────────────>│  Client  │
│  카메라   │               │  (FFmpeg)      │              │          │
└──────────┘               │                │              │  Web:    │
                           │ - RTSP 수신     │              │  HLS.js  │
                           │ - HLS 변환      │              │          │
                           │ - 세그먼트 생성   │              │  iOS:    │
                           │ - 녹화 저장      │              │  AVPlayer│
                           │                │              │          │
                           └────────────────┘              │  Android:│
                                                           │  ExoPlayer│
                                                           └──────────┘
```

### 6.1 HLS 스트리밍 구성
- 세그먼트 길이: 2초
- 플레이리스트 유지: 최근 5개 세그먼트
- 지연시간: 약 4~6초 (준실시간)
- 녹화: 세그먼트를 연결하여 파일로 저장

## 8. 배포 아키텍처

### 8.1 개발 환경
```
Local Development
├── Backend: ts-node-dev (Hot Reload)
├── Web: Vite Dev Server (HMR)
├── DB: Docker Compose (PostgreSQL + Redis)
└── CCTV: 테스트 RTSP 스트림
```

### 8.2 운영 환경 (향후)
```
Production
├── Backend: Docker Container → AWS ECS / EC2
├── Web: Vite Build → S3 + CloudFront (CDN)
├── DB: AWS RDS (PostgreSQL) + ElastiCache (Redis)
├── File: AWS S3
├── Media: EC2 (FFmpeg Media Server)
├── Push: APNs + FCM
└── Monitoring: CloudWatch / Grafana
```
