# 11. 캘린더 외부 동기화 기획

> 작성: v0.14.0 기준 · 상태: **기획(미구현)** · 다음 착수 대기

Project Office의 캘린더 일정을 사용자의 **휴대폰 네이티브 캘린더 / Google Calendar / Outlook** 등 외부 캘린더와 동기화하여, **플랫폼에서 제공하는 알림 대신 OS가 제공하는 알림(푸시·진동·락스크린)**을 활용할 수 있게 한다.

---

## 1. 배경 및 목표

### 문제
- Project Office 캘린더 일정은 앱/웹 내에서만 알림이 울림 (현재는 DB 알림만)
- 사용자가 앱을 열지 않으면 약속 시간을 놓침
- 모바일 앱 FCM/APNs 푸시가 아직 없음 (v0.14.0 기준)

### 목표
| 목표 | 측정 |
|------|------|
| ✅ 휴대폰 OS가 직접 알림을 울림 (앱 미실행 상태에서도) | iOS/Android 락스크린 알림 확인 |
| ✅ 사용자가 여러 기기에서 동일한 일정을 본다 (iPhone + iPad + Mac + 회사 PC) | 일정 동기화 지연 ≤ 1시간 |
| ✅ 프라이버시 보호 (회사 일정이 제3자에게 노출되지 않음) | 토큰 기반 인증 + 폐기 가능 |
| ✅ 구현 난이도 최소화 + 단계적 확장 | Phase 1은 2~3일 내 완성 가능 |

### 비-목표 (이번 기획 범위 외)
- 사내 일정의 완전한 양방향 편집 (외부 앱에서 만든 일정이 Project Office에 역방향 반영) — Phase 2 이후
- 회의실 예약 등 리소스 일정 — 별도 기능

## 2. 기술 선택지 비교

| # | 방식 | 구현 난이도 | UX | 양방향 | 실시간성 | 비고 |
|---|------|-------------|----|--------|----------|------|
| **A** | **ICS 구독 URL** (iCalendar HTTP feed) | **🟢 매우 낮음** | iOS/Android/Google/Outlook 모두 지원 | ❌ 단방향 (읽기) | 📉 15분~24시간 주기 pull | **Phase 1 추천** |
| B | Google Calendar API (OAuth 2) | 🟡 중간 | Google 계정 사용자에겐 완벽 | ✅ | 🟢 실시간 (watch API) | Apple 사용자는 Google 추가 필요 |
| C | CalDAV 서버 구현 | 🔴 높음 | 표준이지만 구현 복잡 | ✅ | 🟡 중간 | 각 플랫폼 호환성 이슈 |
| D | 네이티브 EventKit / CalendarContract | 🟡 중간 | 최고 | ✅ (로컬) | 🟢 즉시 | 모바일 앱 설치 필요 |
| E | FCM/APNs 직접 푸시 | 🟡 중간 | 우리 앱 알림만 가능 | N/A | 🟢 실시간 | OS 캘린더와 무관, 별도 요구 |

### 단계별 추천 경로
1. **Phase 1 (필수 · 2~3일)**: **ICS 구독 URL** — 가장 빠른 시간에 90% 사용자 요구 충족
2. **Phase 2 (선택 · 5~7일)**: **Google Calendar OAuth 양방향 연동** — 실시간성 요구 사용자용
3. **Phase 3 (장기)**: 모바일 앱 내부에서 **EventKit/CalendarContract**로 즉시 동기화 + D와 E 결합

## 3. Phase 1: ICS 구독 URL (권장 MVP)

### 3.1 작동 원리

```
┌───────────────────────┐      ┌──────────────────────────────┐
│  사용자의 iPhone 캘린더   │      │    Project Office 백엔드      │
│  (또는 Google Cal)    │      │                               │
│                       │      │  GET /api/calendar/feed/      │
│  "구독 추가" URL 입력  ├─────▶│       {token}.ics            │
│                       │      │                               │
│  → iOS가 주기적 pull   │      │  → 사용자 일정 조회           │
│    (기본: 매시간 or    │      │  → ICS(iCalendar) 생성        │
│     설정 가능)          │◀─────┤  → VALARM 포함해 응답        │
│                       │      │                               │
│  → 캘린더 앱이 OS      │      │                               │
│    알림을 등록 (앱 X) │      │                               │
└───────────────────────┘      └──────────────────────────────┘
```

**핵심 통찰**: ICS 포맷의 `VALARM` 블록은 **OS 캘린더 앱이 자체 알림으로 등록**한다. 즉 알림이 울리는 것은 OS 몫이고, Project Office는 알림 기능을 직접 만들 필요가 없다.

### 3.2 데이터 모델

```prisma
model CalendarSubscription {
  id              String   @id @default(uuid())
  userId          String   @map("user_id")
  token           String   @unique                           // URL-safe 32자 (crypto.randomBytes)
  name            String                                      // "내 iPhone", "아내 Google" 등 사용자 라벨
  scope           String                                      // "personal" | "personal_dept" | "all"
  includeVacation Boolean  @default(true) @map("include_vacation")  // 휴가 포함 여부
  includeMeeting  Boolean  @default(true) @map("include_meeting")
  includeTasks    Boolean  @default(false) @map("include_tasks")    // 작업지시서 마감일
  reminderMinutes Int[]    @default([10])                    // VALARM 트리거 시간(분 전) 복수
  isActive        Boolean  @default(true) @map("is_active")
  lastAccessedAt  DateTime? @map("last_accessed_at")
  accessCount     Int      @default(0) @map("access_count")
  createdAt       DateTime @default(now()) @map("created_at")
  revokedAt       DateTime? @map("revoked_at")

  user User @relation("CalendarSubscriptions", fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, isActive])
  @@index([token])
  @@map("calendar_subscriptions")
}
```

**보안 요소**:
- `token`은 32바이트 random hex (`crypto.randomBytes(32).toString('hex')`)
- 유출 시 해당 구독만 revoke 가능 (다른 기기 영향 없음)
- `lastAccessedAt` + `accessCount` 로그로 이상 접근 감지
- User delete 시 CASCADE

### 3.3 REST API

| 메서드 | 경로 | 인증 | 설명 |
|--------|------|------|------|
| `POST` | `/api/calendar/subscriptions` | JWT | 새 구독 URL 생성 (기기명·scope·알림시간 지정) |
| `GET` | `/api/calendar/subscriptions` | JWT | 내 구독 목록 |
| `PATCH` | `/api/calendar/subscriptions/:id` | JWT | 이름/scope/알림 변경 |
| `DELETE` | `/api/calendar/subscriptions/:id` | JWT | 구독 폐기 (revoke) |
| `POST` | `/api/calendar/subscriptions/:id/regenerate` | JWT | 토큰 회전 (유출 대응) |
| `GET` | `/api/calendar/feed/:token.ics` | 공개 (토큰 검증) | iCalendar 피드 — **OS가 호출** |

**Feed 엔드포인트 특성**:
- 인증 불필요 (토큰 자체가 인증 증명)
- Content-Type: `text/calendar; charset=utf-8`
- 캐시 헤더: `Cache-Control: private, max-age=900` (15분)
- `If-None-Match` / `ETag`로 304 Not Modified 지원 → 네트워크 절약
- Rate limit: IP당 5분에 60회 (정상 Apple/Google 폴링 대비 여유)

### 3.4 ICS 생성

표준 RFC 5545 준수. 예시:

```ics

VERSION:2.0
PRODID:-//KSCorp//Project Office//KO
CALSCALE:GREGORIAN
METHOD:PUBLISH
X-WR-CALNAME:Project Office - 홍길동
X-WR-TIMEZONE:Asia/Seoul
X-PUBLISHED-TTL:PT15M

BEGIN:VEVENT
UID:event-abc123@po.kscorp.co.kr
DTSTAMP:20260423T020000Z
DTSTART:20260425T010000Z
DTEND:20260425T020000Z
SUMMARY:[회의] 주간 팀 미팅
DESCRIPTION:호스트: 김과장\n참석자: 3명\n회의실: https://po.kscorp.co.kr/meeting/room/abc
LOCATION:대회의실 A
ORGANIZER;CN=김과장:mailto:kim@kscorp.co.kr
STATUS:CONFIRMED
BEGIN:VALARM
TRIGGER:-PT10M
ACTION:DISPLAY
DESCRIPTION:[회의] 주간 팀 미팅 (10분 뒤 시작)

END:VCALENDAR
```

**구현 라이브러리**: Node.js `ical-generator` 또는 `ics` 패키지. 직접 생성도 ~100줄로 가능.

### 3.6 모바일 UI

모바일은 "URL 복사 + 설정 앱 열기 안내" 중심 + **Deep Link**로 iOS의 캘린더 구독 프롬프트 직접 호출 시도 (`webcal://` 스킴).

```
캘린더 설정 → "이 기기 캘린더로 가져오기" 버튼
  → webcal://po.kscorp.co.kr/api/calendar/feed/{token}.ics
  → iOS가 자동으로 "캘린더에 추가하시겠습니까?" 프롬프트
```

### 3.7 보안 고려사항

| 위협 | 대응 |
|------|------|
| 토큰 유출 → 타인이 일정 열람 | 각 구독 개별 revoke + regenerate, 접근 로그 |
| 토큰이 URL에 포함되어 HTTP referer 유출 | HTTPS 강제 + 토큰은 URL path (query string 아님) |
| 민감 정보(기밀 결재 제목 등)가 외부 앱에 노출 | scope별 필터 (예: approval은 제목만, 본문 제외) |
| 대량 크롤링 / DDoS | Rate limit (IP당 5분 60회) + 304 Not Modified |
| 퇴사자가 토큰을 계속 사용 | User.status=inactive 시 구독 자동 비활성 |
| 구독 토큰이 이메일로 공유됨 | 생성 직후 1회만 표시 + 복사 버튼, 재조회 불가 |

## 4. Phase 2: Google Calendar 양방향 연동 (선택)

### 4.1 왜 필요한가
- ICS는 외부 캘린더에서 편집 불가 (읽기 전용)
- iOS 기본 캘린더의 ICS 구독은 **갱신 주기를 사용자가 직접 설정**해야 하고, 기본은 길다
- Google Calendar는 **5분 이내 실시간 반영** 가능

### 4.2 아키텍처

```
┌──────────────────┐         ┌──────────────────────────────────┐
│  사용자 Google 계정│         │  Project Office 백엔드            │
│                  │         │                                  │
│  Google Cal      │◀═══════▶│  googleapis SDK + OAuth2         │
│                  │  watch  │                                  │
│                  │         │  - CalendarExternalSync 모델      │
│  변경 이벤트     │         │  - refresh token (AES-256-GCM)   │
│  ─────────────▶ │         │  - webhook endpoint              │
│  (webhook 호출)  │         │                                  │
│                  │         │                                  │
└──────────────────┘         └──────────────────────────────────┘
```

### 4.3 데이터 모델

```prisma
model CalendarExternalSync {
  id                  String   @id @default(uuid())
  userId              String   @unique @map("user_id")
  provider            String                                 // "google" (추후 "microsoft" 등 확장)
  externalAccountId   String   @map("external_account_id")   // Google 계정 ID
  externalCalendarId  String   @map("external_calendar_id")  // "primary" 또는 ID
  accessTokenEnc      String   @map("access_token_enc")      // AES-256-GCM
  refreshTokenEnc     String   @map("refresh_token_enc")
  tokenExpiresAt      DateTime @map("token_expires_at")
  syncToken           String?  @map("sync_token")            // Google incremental sync
  webhookChannelId    String?  @map("webhook_channel_id")
  webhookExpiresAt    DateTime? @map("webhook_expires_at")
  lastSyncedAt        DateTime? @map("last_synced_at")
  createdAt           DateTime @default(now()) @map("created_at")
  updatedAt           DateTime @updatedAt @map("updated_at")

  user User @relation("CalendarExternalSyncs", fields: [userId], references: [id], onDelete: Cascade)

  @@index([provider])
  @@map("calendar_external_syncs")
}
```

### 4.4 동기화 흐름

**Push (Project Office → Google)**
- CalendarEvent `create/update/delete` → googleapis SDK로 대응 이벤트 생성
- Event ID 매핑 테이블로 중복 방지

**Pull (Google → Project Office)**
1. 초기: 사용자 연동 시 Google의 `events.list` + `syncToken` 저장
2. 증분: Google **watch** 구독으로 webhook 수신
3. Webhook 수신 시 `events.list(syncToken=...)` 호출 → 변경분만 반영

**충돌 해결**:
- `updatedAt` 비교 → 최신이 우선
- 동시 편집 충돌 → "외부 수정을 Project Office에 반영"이 기본

### 4.5 OAuth 구성

| 항목 | 값 |
|------|----|
| Scope | `https://www.googleapis.com/auth/calendar.events` |
| Consent | Google Cloud Console에서 OAuth 앱 등록 |
| Redirect URI | `https://po.kscorp.co.kr/api/calendar/google/oauth/callback` |
| 토큰 저장 | AES-256-GCM 암호화 (mailCrypto 재사용 가능) |

### 4.6 리스크

- Google API quota (기본 10만 req/일) — 대규모 조직에서 한도 상향 필요
- Google 측 변경으로 webhook 만료 → cron으로 갱신 필요
- OAuth 동의 화면 Google 검증 필요 (민감 scope)

## 5. Phase 3: 모바일 네이티브 통합 (장기)

### 5.1 EventKit (iOS) + CalendarContract (Android)

앱 내 설정에서 "기기 캘린더에 저장하기" 토글 → 앱이 직접 OS API 호출해서 로컬 이벤트 생성.

**장점**
- 네트워크 불필요 (이미 앱이 알고 있음)
- 알림이 즉시 등록
- 사용자가 기기 캘린더 UI에서 바로 확인

**단점**
- 동기화는 단방향 (앱 → 기기)
- 기기별로 개별 설정 (여러 기기를 쓰면 반복)

**구현**
- iOS: `react-native-calendar-events` 또는 Expo `expo-calendar`
- Android: 동일 라이브러리

## 8. 의사결정 포인트 (사용자 선택 필요)

1. **Phase 1만? 아니면 Phase 2까지?**
   - Phase 1만: 2~3일, 읽기 전용, 동기화 15분~1시간 지연
   - Phase 1+2: 총 8일 내외, 양방향, 실시간

2. **ICS scope 기본값**
   - (A) 개인 일정만 (최소) — 권장
   - (B) 개인 + 참석 확정 회의 — 사용자 다수가 원할 가능성
   - (C) 모든 일정 — 필요한 사용자만 변경

3. **휴가는 ICS에 포함?**
   - 본인 휴가는 포함이 자연스러움
   - 타인 휴가(부서 캘린더 볼 때)는 포함할지, 이름만/전체 공개 여부

4. **기밀 결재 내용**
   - SUMMARY에 결재 제목 포함? (타인이 보면 민감할 수 있음)
   - 권장: ICS는 **일정 위주**로만 구성, 결재는 제외하거나 "결재 진행 중" 같은 일반 문구만

5. **구독 URL 만료**
   - 영구? (revoke 전까지)
   - 90일 자동 만료 + 재생성 필요? (보안 강화)

## 10. 위험 관리

| 위험 | 완화책 |
|------|--------|
| ICS 갱신 주기가 길어 사용자가 "못 받았다"고 불만 | UI에 "iPhone은 기본 1시간, Google은 6시간. 설정에서 조정 가능" 명시 |
| 토큰 유출로 일정 정보 공개 | 개별 revoke + 접근 로그, 의심 접근 시 알림 |
| 대량의 일정이 ICS에 포함되면 응답 느림 | 과거 30일 + 미래 365일로 범위 제한 |
| 타임존 처리 오류 | 서버는 항상 UTC, ICS는 Asia/Seoul 명시 |
| 퇴사자 일정 계속 sync | User.status=inactive 시 cron이 모든 구독 revoke |
