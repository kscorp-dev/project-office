# 기능 명세서: 작업지시서 (작업명세서)

> TOPAZ (topaz.taekeon.com) 시스템의 작업지시서 기능을 참고하여 설계.
> 범용 그룹웨어에 맞게 확장 및 일반화함.

## 1. 개요

업무 지시, 작업 내용, 진행 상황을 체계적으로 관리하는 모듈이다.
작업지시서를 작성하여 담당자에게 전송하고, 6단계 워크플로우로 진행 상황을 추적한다.
TOPAZ의 작업지시서 구조를 기반으로, 발주처 관리, 시공 연동, 대금청구 등의 기능을 포함한다.

## 2. TOPAZ 시스템 분석 (참고)

### 2.1 TOPAZ 메뉴 구조
```
작업지시서
├── 작업지시서 (목록/등록/상세)
├── 작업지시서[임시/확인]
└── 시공지시서

차량 출고일지
└── 출고일지(관리자)

발주/입고 관리
└── 발주/입고 확인

시스템관리
├── 발주 업체 정보관리
├── 배송지 정보관리
└── 비밀번호 변경
```

### 2.2 TOPAZ 작업지시서 필드 구조
| 영역 | 필드 |
|------|------|
| 기본 정보 | 작업명칭, 작업지시서번호(자동채번), 작업지시일, 작업완료요청일 |
| 담당자 | 작성자, 영업지원, 영업담당, 디자인담당, 출력담당, 공장장 |
| 사고차 여부 | 사고차 체크박스 |
| 발주처 | 상호, 더존업체코드, 담당자, 연락처, 계산서이메일 |
| 대금청구 | 세금계산서/기타, 청구액, 부가세포함/부가세별도 |
| 미청구시 사유 | A/S, 샘플, 선입금 완료, 작업자 실수 재작업, 기타(보험청구 등) |
| 실제공방회사 | 회사명, 연락처 |
| 생산 정보 | 프린터 종류, 출력 품질, 해상도(DPI), 온도(℃), 습도(%) |
| 작업 상세 | 차종명, 세부작업명, 수량, 단위, 출력원단, LOT No., 출력 프로파일 |
| 작업 이미지 | 작업용 이미지(ai) 첨부, 참고 이미지 |
| 시공 | 시공필요여부, 시공업체명 |
| 배송지 | 회사명, 담당자명, 연락처, 주소 |
| 기타 | 추가교청사항, 택배신용, 제조담당 Comment |

### 2.3 TOPAZ 워크플로우 (6단계)
```
작업지시 → 출력완료 → 부분시공완료 → 작업(시공)완료 → 대금청구완료 → 최종완료
```
- 각 단계 전환 시 작성일, 작성자, 내용이 기록됨
- "폐기" 상태도 존재

### 2.4 TOPAZ 검색 필터
- 작업명칭, 발주처명
- 작업지시일자 (기간)
- 대금청구필요여부, 시공필요여부, 처리상태
- 영업담당자, 영업지원담당자, 디자인담당자

## 3. Project Office 작업지시서 설계

### 3.1 작업지시서 작성

**사용자 시나리오:**
1. "작업지시서 작성" 진입
2. 작업 기본 정보 입력 (제목, 내용, 기한)
3. 담당자 지정 (역할별 복수 지정 가능)
4. 발주/고객 정보 입력 (선택)
5. 작업 상세 항목 추가 (품목, 수량, 단위 등)
6. 첨부파일/이미지 추가
7. 대금청구 정보 설정 (선택)
8. "임시저장" 또는 "작업지시" 전송

**상세 요구사항:**

### 3.5 워크플로우 (6단계 + 확장)

**TOPAZ의 6단계를 기반으로 범용 확장:**

```
[임시저장] ──전송──> [작업지시] → [진행중] → [부분완료] → [작업완료] → [대금청구완료] → [최종완료]
                      │                                                    │
                      └── [폐기]                                           └── [대금미청구] → [최종완료]
```

**상태 정의:**

| 상태 | 코드 | 설명 | 다음 가능 상태 |
|------|------|------|-------------|
| 임시저장 | draft | 작성 중, 미전송 | 작업지시, 폐기 |
| 작업지시 | instructed | 담당자에게 전송됨 | 진행중, 폐기 |
| 진행중 | in_progress | 작업 수행 중 | 부분완료, 작업완료 |
| 부분완료 | partial_complete | 일부 작업 완료 | 작업완료 |
| 작업완료 | work_complete | 전체 작업 완료 | 대금청구완료, 최종완료 |
| 대금청구완료 | billing_complete | 대금 청구 처리 완료 | 최종완료 |
| 최종완료 | final_complete | 모든 프로세스 종료 | - |
| 폐기 | discarded | 취소/폐기 | - |

**각 단계 전환 시 기록:**
- 전환 일시
- 전환 실행자
- 코멘트 (메모)

### 3.6 디자인 파일 관리 (Design File Workflow)

> 디자인 팀이 작업지시서에 디자인 파일(AI 등)을 업로드하고,
> 공장(생산팀)이 다운로드하면 로그가 기록되며, 진행 순서에 반영된다.

**워크플로우:**
```
[디자인팀]                    [시스템]                   [공장/생산팀]
    │                           │                           │
    ├─ AI 파일 업로드 ─────────▶│                           │
    │                           ├─ 업로드 로그 기록          │
    │                           ├─ 담당자에게 알림 전송 ────▶│
    │                           │                           │
    │                           │◀── 파일 다운로드 ─────────┤
    │                           ├─ 다운로드 로그 기록        │
    │                           ├─ 상태: "디자인 확인" 자동 │
    │  ◀── 다운로드 확인 알림 ──┤   기록 (진행 순서에 반영)  │
    │                           │                           │
    │                           │◀── 작업 시작/완료 보고 ───┤
    │                           ├─ 진행 이력 기록            │
    └───────────────────────────┘                           │
```

**상세 요구사항:**

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| DESIGN-001 | 디자인 파일 업로드 (AI, PSD, PDF, EPS, SVG 등) | 필수 |
| DESIGN-002 | 파일 버전 관리 (v1, v2... 수정본 업로드 시 버전 자동증가) | 필수 |
| DESIGN-003 | 업로드 시 자동 로그 기록 (업로더, 시각, 파일정보) | 필수 |
| DESIGN-004 | 업로드 완료 시 작업 담당자(공장)에게 푸시 알림 | 필수 |
| DESIGN-005 | 파일 다운로드 시 로그 자동 기록 (다운로더, 시각, IP) | 필수 |
| DESIGN-006 | 최초 다운로드 시 진행 순서에 "디자인 파일 확인" 이력 자동 기록 | 필수 |
| DESIGN-007 | 파일별 다운로드 횟수/이력 조회 (누가 언제 받았는지) | 필수 |
| DESIGN-008 | 디자인 파일 미리보기 (썸네일 자동 생성) | 중요 |
| DESIGN-009 | 디자인 수정 요청 코멘트 (공장→디자인팀) | 중요 |
| DESIGN-010 | 디자인 확정(승인) 처리 (공장에서 확인 완료 표시) | 중요 |
| DESIGN-011 | 파일 접근 권한 제어 (지시서 관련자만 다운로드) | 필수 |
| DESIGN-012 | 대용량 파일 지원 (AI 파일 최대 200MB) | 필수 |

**파일 로그 기록 항목:**

| 로그 유형 | 기록 내용 |
|----------|----------|
| upload | 업로더 ID, 파일명, 파일크기, 버전, 업로드 시각 |
| download | 다운로더 ID, 파일명, 버전, 다운로드 시각, IP 주소, 디바이스 |
| view | 조회자 ID, 파일명, 조회 시각 (미리보기) |
| approve | 확인자 ID, 파일명, 버전, 확정 시각, 코멘트 |
| reject | 반려자 ID, 파일명, 버전, 반려 시각, 수정요청 사유 |

**진행 순서 자동 기록 규칙:**
1. 디자인 파일 업로드 → task_status_history에 "디자인 파일 등록" 이벤트 기록
2. 공장 담당자 최초 다운로드 → "디자인 파일 확인" 이벤트 자동 기록
3. 디자인 확정(승인) → "디자인 확정" 이벤트 기록
4. 수정 요청 시 → "디자인 수정 요청" 이벤트 기록 + 디자인팀 알림
5. 수정본 재업로드 → "디자인 수정본 등록 (v2)" 이벤트 기록

## 6. 비즈니스 규칙

1. 작업지시서번호 형식: `T{YYYYMMDD}{5자리순번}` (예: T2026040800001)
2. 작업지시 상태 이후에는 삭제 불가 (폐기만 가능)
3. 상태 전환은 순차적으로만 가능 (건너뛰기 불가, 단 대금미청구 시 작업완료 → 최종완료 가능)
4. 각 상태 전환 시 이력이 자동 기록된다
5. 폐기 시 사유를 반드시 작성해야 한다
6. 마감일 초과 작업은 목록에서 "지연" 표시 (D+ 표기)
7. 체크리스트 진행률 = (완료 항목 수 / 전체 항목 수) × 100%
8. 대금청구 불필요 설정 시 미청구 사유 필수
9. 거래처 정보는 재사용 가능 (검색/선택)
10. 작업지시서 인쇄 시 A4 양식에 맞춰 출력
11. 엑셀 다운로드 시 검색 결과 전체 포함
12. 디자인 파일은 작업지시서 관련자(지시자, 담당자)만 다운로드 가능
13. 디자인 파일 업로드 시 이전 버전은 is_latest=false로 변경 (버전 이력 유지)
14. 디자인 파일 최대 200MB, 허용 확장자: ai, psd, pdf, eps, svg, png, jpg, tiff
15. 공장 담당자의 최초 다운로드 시 자동으로 진행 이력에 "디자인 파일 확인" 기록
16. 디자인 파일 다운로드/업로드/조회 모든 행위는 task_file_logs에 기록
17. 디자인 확정 후 수정본 재업로드 시 확정 상태 초기화 (재확인 필요)
