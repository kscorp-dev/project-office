# 푸시 알림 운영 가이드

마지막 갱신: 2026-04-30

Project Office 의 모바일 푸시 알림이 실제로 어떻게 동작하는지, 무엇을 설정하고
무엇을 확인해야 하는지 정리.

## 1. 아키텍처

```
모바일 앱
  └─ usePushNotifications 훅
      ├─ Expo push token 발급
      └─ POST /api/notifications/devices ──→ Backend (Postgres UserDevice)
                                                    │
이벤트 발생 (결재/메시지/회의 ...) ─→ createNotification()
                                          ├─ DB Notification insert
                                          ├─ WebSocket emit (실시간)
                                          └─ sendPushToUser() ──→ Expo Push API
                                                                       │
                                                          APNs (iOS) / FCM (Android)
                                                                       │
                                                                  실기기
```

## 2. 환경변수 (백엔드 `.env`)

| 변수 | 용도 | 미설정 결과 |
|-----|------|-----------|
| `EXPO_ACCESS_TOKEN` | Expo Push API 인증 토큰 | rate limit 분당 ~600 (운영 부적합) |
| `DISABLE_PUSH` | `true` 면 발송 스킵 | 실서비스에서 미설정. 테스트/CI 만 사용 |

발급 방법:
1. https://expo.dev/accounts/<account>/settings/access-tokens
2. "Create" → 권한은 push notifications 만 체크
3. 발급된 토큰을 `.env` 의 `EXPO_ACCESS_TOKEN` 에 입력

## 3. 환경변수 (모바일 `eas.json`)

| 프로파일 | EXPO_PUBLIC_APS_ENV | 용도 |
|---------|---------------------|------|
| development / development-simulator | development | dev client, 사내 테스트 |
| preview | production | TestFlight 직전 |
| production | production | App Store 배포 |

`with-aps-environment.js` config plugin 이 빌드 시점에 entitlement 분기.
잘못된 환경 토큰으로 발송하면 silent fail (로그만 남고 알림 안 옴).

## 4. 헬스체크 (운영자가 가장 먼저 봐야 할 것)

### 4-1. 백엔드 부팅 로그
서버 부팅 후 1.5초 뒤 `📱 Push notifications health` 로그가 한 번 출력된다:
```
{
  enabled: true,
  hasAccessToken: true,
  devices: { ios: 12, android: 8, total: 20 }
}
```

경고는 별도 line 으로 `[push] warning` 출력:
- `EXPO_ACCESS_TOKEN 미설정 — Expo rate limit 엄격`
- `⚠️ PRODUCTION 환경에서 EXPO_ACCESS_TOKEN 가 비어있음`
- `등록된 활성 디바이스 0개`

### 4-2. 관리자 API
```bash
GET /api/admin/push/health   # 권한: admin / super_admin
```
응답 페이로드는 위 구조체 그대로. 모니터링 시스템에서 5분마다 체크하면
디바이스 수 급감을 빠르게 감지 가능.

### 4-3. 모바일 관리자 콘솔
관리콘솔 화면(`/admin`) 하단의 "📲 테스트 푸시 발송" 버튼 →
본인 기기로 즉시 푸시 1건. 알림이 도착하면 발송 경로 OK.

## 5. 푸시 페이로드 매핑 (참고)

`mapToMobilePayload` 가 DB enum 을 모바일이 이해하는 단순 키로 변환.

| DB type | mobile data.type | 잠금화면 인라인 액션 (categoryId) |
|---------|-----------------|----------------------------------|
| approval_pending | approval | approval ([승인]/[반려]) |
| approval_approved/rejected/recalled/reference | approval | (없음) |
| message_received / message_mention | message | message ([답장]/[읽음]) |
| mail_received | mail | (없음) |
| meeting_invited | meeting | meeting ([수락]/[거절]) — meta.ring=true 시 CallKit UI |
| meeting_starting_soon / meeting_minutes_ready | meeting | (없음) |
| task_assigned / task_status_changed | task | (없음) |
| vacation_* | vacation | (없음) |
| post_must_read 등 | post | (없음) |

## 6. CallKit 즉시 호출 (meeting_invited + meta.ring=true)

호스트가 회의 진행 중 "📞 호출" 버튼 누르면 `POST /meeting/:id/ring` 호출 →
참가자 푸시 페이로드에 `data.ring='1'` 포함됨.

모바일은 포그라운드 도착 시 `displayIncomingMeetingCall()` 트리거 → 풀스크린
CallKit UI 출현. 잠금화면이면 OS 가 인라인 [수락]/[거절] 버튼만 보여줌.

자세한 검증은 [모바일 빌드 가이드](모바일-빌드-가이드.md#2-callkit-동작-검증-ios-실기기-전용) 참고.

## 7. 트러블슈팅

| 증상 | 원인 후보 | 확인 방법 |
|-----|----------|----------|
| 푸시가 전혀 안 옴 | DISABLE_PUSH=true | `GET /admin/push/health` 의 enabled 확인 |
| iOS 만 안 옴 | aps-environment 환경 불일치 | EAS 빌드의 EXPO_PUBLIC_APS_ENV 검토 |
| Android 만 안 옴 | FCM 키 미발급 / package id 불일치 | Expo 대시보드의 FCM credentials 확인 |
| 잠금화면 인라인 버튼 안 보임 | iOS 14 미만 / categoryId 미설정 | `notification.service.ts` mapToMobilePayload 검사 |
| CallKit UI 안 뜸 | UIBackgroundModes.voip 누락 | `npm run verify:build` 의 voip 항목 |
| "ExponentPushToken not valid" | invalidTokens 처리 | push.service 가 자동으로 `isActive=false` 로 마킹 |
| 일부 사용자만 안 옴 | 디바이스 토큰 만료 / 앱 재설치 후 미등록 | UserDevice 테이블에서 `isActive=true` + 최근 lastUsedAt 확인 |
